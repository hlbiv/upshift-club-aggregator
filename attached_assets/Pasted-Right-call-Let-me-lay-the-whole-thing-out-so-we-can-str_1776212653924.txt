## Conventions (locked first)

- **TS camelCase ↔ SQL snake_case**, matching existing style (`clubNameCanonical` ↔ `club_name_canonical`).
- **Timestamps:** `scrapedAt` (event of record), `firstSeenAt` / `lastSeenAt` (where change detection matters), `createdAt` / `updatedAt` (only for records humans touch, e.g. admin overrides).
- **Polymorphic refs:** `(entity_type, entity_id)` pattern for cross-entity tables (scrape_health, scrape_run_logs). No nullable-FK-per-type bloat.
- **Natural keys = unique constraints** on every scraped table for idempotent upserts. Explicit, named.
- **Check constraints** for enums (avoid Postgres native enums — harder to migrate).
- **`player_name` is text, never an FK.** Aggregator doesn't have player identity — that's upshift-player-platform's job. Cross-roster attribution is a fuzzy derived job.
- **Aggregator stops at "identified scraped fact."** Coach effectiveness, roster churn, movement events are **derived** tables (materialized views or nightly jobs), not scraped directly.

## Cross-domain FK map (the critical joins)

| Join | Path | Purpose |
|---|---|---|
| Club → career | `canonical_clubs ← coach_career_history (entity_type='club')` | "who has coached here" |
| College → career | `colleges ← coach_career_history (entity_type='college')` | coach college stints |
| Coach → placements | `coach_career_history.entity_id (club) → clubRosterSnapshots.player_name → collegeRosterHistory.player_name` (fuzzy) | the moat: D1 placement rates |
| Event → results | `events → eventSchedule` OR `events → matches (event_id)` | two ways — eventSchedule is tournament-internal, matches is the superset |
| Club → match history | `canonical_clubs ← matches.home_club_id/away_club_id` | season records |

## Decisions I'd flag for approval

1. **Drop existing `clubEvents` table.** It's empty, wrong-shaped, and `eventTeams` replaces it. Cleaner than renaming.

2. **`matches.platform_match_id` is the dedup key, not `(home_club, away_club, date)`.** Because league vs tournament matches may both exist with same teams on same date; platform IDs disambiguate. Partial unique index when platform_match_id is not null; accept dupes for manually-entered matches.

3. **`coachEffectiveness` is derived, not scraped.** The whole table regenerates nightly from career × college rosters. No incremental updates. Simpler; correctness > performance at this scale.

4. **`matches` is the superset of `eventSchedule`.** `eventSchedule` is redundant if matches has `event_id`. **I'd drop `eventSchedule` and just use matches everywhere.** Spec had both; I'd unify. Flag for you.

5. **Coach `person_hash` as the dedup identity.** Because one coach moves between clubs — you want `coach_movement_events` to actually link them. `person_hash` solves that without requiring a separate `coaches` master table (which would be another migration).

6. **No aggregator-side player identity.** Player rosters are text names. Cross-matching is fuzzy in jobs, not structural. Pushes player identity to upshift-player-platform where it belongs.

## What I think is still loose

- **Does `coachDiscoveries` stay as "the coach" or do we introduce a proper `coaches` master?** Current: it's both the scraped row AND the identity. That's fine if we accept `person_hash` as dedup, but if one real coach ever has 5 `coach_discoveries` rows across 5 clubs, then `coachCareerHistory` FKs need to point at one canonical coach, not any one of the discoveries. **This is the biggest structural question.** My lean: add `coaches` master table (minimal: id, person_hash, display_name, primary_email, first_seen_at, last_seen_at), and `coachDiscoveries` gets a `coach_id` FK pointing at it. Small cost, clean.

- **Event team matching to `canonical_clubs`.** `eventTeams.canonical_club_id` is nullable — how does the matching happen? Online during scrape (fuzzy match at write time) or offline batch? I'd say **offline batch** — scrape writes team_name_raw with null club_id, a linker job fills it in. Matches the aggregator's existing deferred-enrichment pattern.

- **`matches` uniqueness when `platform_match_id` is null.** Manually-entered / partial scraped matches. Options: (a) accept dupes, (b) composite unique `(home_team_name, away_team_name, match_date, age_group, gender)`. Lean (b) with `WHERE platform_match_id IS NULL` partial index, so scraped and manual coexist.

## My recommended tightenings before migration

1. Add `coaches` master table, refactor `coachDiscoveries` → `coach_id` FK. Raises total new tables from ~14 to 15.
2. Drop `eventSchedule`, use `matches` everywhere.
3. Drop `clubEvents` (empty, wrong shape).
4. Commit to "polymorphic `(entity_type, entity_id)` with app-layer validation" as the pattern for `scrapeHealth`, `coachCareerHistory`, `coachMovementEvents`, `scrapeRunLogs`.
