# Upshift Club Aggregator

A scraper toolkit, PostgreSQL graph database, and REST API for US youth soccer club data. Covers 127 league directories across all tiers of the US youth soccer pyramid — from MLS NEXT down to 54 state associations — normalizing and deduplicating clubs into a single canonical dataset.

---

## What It Does

1. **Scrapes** club rosters from 115 publicly accessible league directories (7 Tier-1 national, 13 Tier-2 high performance, 41 Tier-3 regional, 54 Tier-4 state associations)
2. **Normalizes** club names (stripping suffixes, fixing casing) and **deduplicates** across sources with fuzzy matching (RapidFuzz, threshold 88)
3. **Enriches** clubs with website URLs — first from directory pages, then via Brave Search API for remaining clubs
4. **Discovers coaches** by scraping staff pages (SportsEngine, LeagueApps, WordPress)
5. **Seeds** a PostgreSQL graph database with canonical clubs, aliases, affiliations, events, and coaches
6. **Exposes** a typed REST API on port 8080 with search, analytics, and graph traversal endpoints

---

## Reliability

- **Retry logic**: All HTTP requests use utils/retry.py — exponential backoff (2s base, cap 60s), 3 retries. ConnectionError, Timeout, 5xx → TransientError. Playwright navigation errors retried similarly.
- **Failure reporting**: run.py tracks each league scrape result. End-of-run summary shows counts by FailureKind: timeout, network, parse_error, zero_results, unknown.
- **Multi-state events**: GotSport events spanning multiple states (MN, WV) set multi_state=true in config; clubs keep blank state rather than inheriting wrong parent state.
- **Deduplication**: RapidFuzz token-sort ratio at threshold 88 merges near-identical club names across sources into a single canonical record.

The two gotchas

2. "Games" means two different things.

In the aggregator's world: team-vs-team fixtures (home/away, scores, dates). Pure reference data.
In Upshift's world: player-tagged appearances (game_players table), jersey numbers, highlight clips, stat credit.

Three design calls I'm making that you should sanity-check before I write:

Two tables, not one. scrape_run_logs = append-only history of every run. scrape_health = current-state rollup per entity. You want both — logs for debugging, health for "show me stale clubs" queries. Could be one table but separation keeps queries fast.
entityId is a bare integer, not FK. Because it references multiple tables (clubs, colleges, leagues, coaches). Polymorphic. The alternative is nullable FK columns per entity type — messier. Runtime validation in the API layer.
failureKind mirrors Python FailureKind exactly so the Python side just stringifies the enum. No translation layer.

Decisions I'd flag for approval
Drop existing clubEvents table. It's empty, wrong-shaped, and eventTeams replaces it. Cleaner than renaming.
matches.platform_match_id is the dedup key, not (home_club, away_club, date). Because league vs tournament matches may both exist with same teams on same date; platform IDs disambiguate. Partial unique index when platform_match_id is not null; accept dupes for manually-entered matches.
coachEffectiveness is derived, not scraped. The whole table regenerates nightly from career × college rosters. No incremental updates. Simpler; correctness > performance at this scale.
matches is the superset of eventSchedule. eventSchedule is redundant if matches has event_id. I'd drop eventSchedule and just use matches everywhere. Spec had both; I'd unify. Flag for you.
Coach person_hash as the dedup identity. Because one coach moves between clubs — you want coach_movement_events to actually link them. person_hash solves that without requiring a separate coaches master table (which would be another migration).
No aggregator-side player identity. Player rosters are text names. Cross-matching is fuzzy in jobs, not structural. Pushes player identity to upshift-player-platform where it belongs.
What I think is still loose
Does coachDiscoveries stay as "the coach" or do we introduce a proper coaches master? Current: it's both the scraped row AND the identity. That's fine if we accept person_hash as dedup, but if one real coach ever has 5 coach_discoveries rows across 5 clubs, then coachCareerHistory FKs need to point at one canonical coach, not any one of the discoveries. This is the biggest structural question. My lean: add coaches master table (minimal: id, person_hash, display_name, primary_email, first_seen_at, last_seen_at), and coachDiscoveries gets a coach_id FK pointing at it. Small cost, clean.
Event team matching to canonical_clubs. eventTeams.canonical_club_id is nullable — how does the matching happen? Online during scrape (fuzzy match at write time) or offline batch? I'd say offline batch — scrape writes team_name_raw with null club_id, a linker job fills it in. Matches the aggregator's existing deferred-enrichment pattern.
matches uniqueness when platform_match_id is null. Manually-entered / partial scraped matches. Options: (a) accept dupes, (b) composite unique (home_team_name, away_team_name, match_date, age_group, gender). Lean (b) with WHERE platform_match_id IS NULL partial index, so scraped and manual coexist.
